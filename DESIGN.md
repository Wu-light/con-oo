
一、整体架构
代码采用三层分层架构：
Svelte 组件层：包含 Board、Keyboard、Actions、Header、Modal 等核心组件，通过 Svelte 存储读取状态、调用存储方法，是直接的交互层。
Store Adapter 层：对应文件 src/node_modules/@sudoku/stores/grid.js，内部持有游戏领域对象，对外的 Svelte 可写存储。
领域对象层：对应文件 domain/Game.js 和 Sudoku.js，负责核心游戏逻辑实现，是规则的载体。


二、领域对象设计

1. Sudoku
核心职责：
维护 9x9 棋盘数据
提供格子修改、棋盘数据获取、实例拷贝、数据序列化的基础能力

2. Game
核心职责：
持有 Sudoku 实例，管理游戏操作历史
提供猜数、撤销、重做的统一操作接口
支持撤销 / 重做状态查询和游戏状态序列化
历史管理策略：采用快照模式管理操作历史，以数组存储历史快照，通过索引定位当前状态。初始状态生成首个快照，每次猜数会截断重做分支并新增快照；撤销和重做通过索引切换快照，恢复对应棋盘状态，符合标准的撤销重做交互逻辑。


三、领域对象与 View 交互
1. View 直接对象
View 不直接操作领域对象，仅消费 Store Adapter 层。Adapter 封装 Game 实例，对外暴露响应式状态和操作方法。
2. View 层获取数据
原始谜题棋盘：只读状态，用于区分预填格子
用户当前棋盘：核心渲染数据，支持操作
冲突单元格：自动派生的错误格子列表，用于高亮提示
撤销 / 重做状态：布尔值，控制按钮禁用状态
游戏完成状态：自动派生，判断游戏是否通关
3. 用户操作流转流程
用户的所有操作（输入数字、撤销、重做、使用提示、新建游戏），均由组件触发 Adapter 层方法，Adapter 调用领域对象完成逻辑处理，最终同步状态到 View 层。
4. UI 自动更新机制
领域对象状态变更后，Adapter 层的 syncFromDomain 函数会从领域对象获取最新深拷贝数据，调用 Svelte 存储的赋值方法。Svelte 检测到存储值的引用变化，自动通知订阅组件，触发界面重新渲染。


四、响应式机制说明
1. 依赖的 Svelte 响应式能力
基于 Svelte 核心响应式机制实现：使用可写存储管理状态，派生存储自动计算衍生数据，通过存储订阅语法实现组件自动更新，自定义存储满足 UI 消费规范。
2. 暴露给 UI 的响应式数据
用户棋盘、原始谜题、冲突格子、撤销 / 重做状态均为响应式数据，实时同步到界面。
3. 领域对象内部隐藏状态
游戏历史快照数组、历史索引、Sudoku 内部棋盘引用均为内部状态，不直接暴露给 UI，仅通过封装后的状态间接体现。
4. 直接修改内部对象的问题
Svelte 响应式基于赋值引用检测，直接修改数组 / 对象内部元素不会触发界面更新。本方案通过 Adapter 层统一赋值新引用，彻底规避该问题。


五、代码改进说明

明确分层职责，解决原有逻辑分散问题
所有状态变更经过领域对象，保证状态一致性
